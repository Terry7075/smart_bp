摘要

本報告為專題「明德 e 達人」之個人負責成果，著重說明本人實作之柑仔店物資訂購（需求單）與智慧小幫手兩大模組。App 整體採 Flutter 開發、Supabase 為後端；登入、健康藥單、志工儀表板等由團隊其他成員完成，本稿僅在必要處說明介接方式，不展開非本人模組之設計細節。

物資訂購模組讓長輩在 App 內瀏覽商品、建立代購需求單並寫入雲端 orders、order_items，可查詢個人訂單狀態，並串接全聯 pxbox 外部搜尋作價格參考。志工端可讀取需求清單並更新處理進度（RLS 權限由團隊共用 schema 定義）。

小幫手模組提供長輩友善對話介面：以意圖分類區分閒聊、查代購進度、App 帶路教學；查詢時讀取 Supabase 真實訂單資料，回覆採輕鬆口語而非制式客服稿；支援語音輸入與即時字幕、對話歷史雲端同步。不依賴付費 OpenAI，正式環境以規則引擎為主。

關鍵詞：物資訂購、需求單、小幫手、Flutter、Supabase、高齡友善、Riverpod

第一章 緒論
════════════════════════════════════════

1.1 研究背景

社區長輩常需請志工或親友協助採買日用品，但一般電商 App 介面複雜、購物車與促銷資訊過多，不利長輩獨立操作。同時，長輩對智慧手機的問答、查進度需求增加，若僅依賴電話或 LINE，志工負擔重、資訊也容易斷裂。

本專題 App「明德 e 達人」為團隊共同開發之社區服務平台。本人負責其中兩項與長輩日常最相關的能力：簡化之物資需求單（柑仔店）與內建小幫手（查代購、陪聊、帶路）。

1.2 研究動機

（1）訂購流程過繁：長輩只需告訴社區要買什麼，不需完整電商結帳。
（2）進度不透明：送出需求後常不知道志工處理到哪一步。
（3）操作說明不足：不熟悉 App 時需要口語引導與一鍵前往正確頁面。
（4）成本限制：小幫手不宜依賴付費 GPT 或 24 小時自架 AI 主機。

1.3 研究目的

（1）實作長輩端物資需求單建立、列表與狀態查詢（Supabase 持久化）。
（2）實作小幫手：自動判斷閒聊或查系統、查詢真實代購紀錄、輕鬆語氣回覆、可帶路至柑仔店／訂單頁。
（3）提供語音輸入與即時字幕、對話歷史（雲端＋本機備援）。
（4）與團隊共用登入、首頁入口、志工端讀單流程介接。

1.4 研究範圍

【本報告詳述（本人開發）】
・柑仔店商品列表、購物車式需求單、送出訂單
・長輩訂單紀錄頁（/shop/orders）
・商品圖、全聯 pxbox 外部連結
・小幫手對話、意圖、語音、歷史
・orders、order_items、assistant_chat_sessions 相關 App 端

【團隊其他成員／僅簡述介接】
・登入註冊、角色分流
・健康藥單 OCR、志工藥單任務
・交通／學習／活動分頁（占位）
・志工儀表板 UI（代購列表由本人 repository 提供資料層）
・吃藥提醒、通知排程

1.5 報告架構

第二章為技術背景；第三至五章集中於物資訂購與小幫手之需求、設計與實作；第六章為測試與成果；第七章為結論。

第二章 相關技術
════════════════════════════════════════

2.1 高齡友善設計（本模組採用）

・柑仔店、小幫手頁面：大字標題、大按鈕、高對比（綠／棕／奶油色）。
・錯誤訊息口語化（如「暫時讀不到您的資料」）。
・小幫手常見問題：點一下即可送出，減少打字。

2.2 技術選型（與本模組相關）

Flutter + Riverpod …… 柑仔店 UI、訂單狀態、小幫手對話狀態
GoRouter ………………… /shop、/shop/orders、/assistant
Supabase ………………… orders、order_items、assistant_chat_sessions
speech_to_text …………… 小幫手語音與即時字幕
shared_preferences ……… 對話歷史離線備援
url_launcher ……………… 開啟全聯 pxbox 參考價格頁

2.3 與一般電商 App 之差異

交易模式
　一般電商：線上金流、配送追蹤
　本專題：需求單，志工現場比對採買

商品來源
　一般電商：自建庫存
　本專題：Supabase 商品表＋圖片資產

價格
　一般電商：結帳金額
　本專題：參考全聯官網，實際以門市為準

售後查詢
　一般電商：訂單編號
　本專題：小幫手口語查「代購到哪了」

3.4 使用案例

【案例一：長輩送出物資需求單】
1. 長輩在柑仔店選商品、調整數量
2. 確認送出需求單
3. ShopPage 呼叫 ShopOrdersRepository.createOrderWithItems()
4. 寫入 Supabase 的 orders 與 order_items，狀態為 pending
5. App 提示長輩已送出

【案例二：小幫手查代購進度】
1. 長輩問「代購到哪了」
2. AssistantIntent 判斷為 systemData（查系統資料）
3. AssistantSnapshotLoader 向 Supabase 查詢個人訂單
4. 以輕鬆語氣回覆狀態，並顯示「查看代購紀錄」按鈕

【案例三：小幫手帶去柑仔店】
1. 長輩問「怎麼用柑仔店」
2. 判斷為 appGuide（教學帶路）
3. 顯示操作步驟與「前往柑仔店」按鈕
4. 點擊後 GoRouter 導向 /shop，開啟 ShopPage

第四章 系統設計
════════════════════════════════════════

4.1 本人模組在整體架構中的位置

【本人負責】
・shop/presentation … 柑仔店畫面、我的需求
・shop/data … ShopOrdersRepository、商品 Repository
・assistant/presentation … 對話頁、語音
・assistant/data … 意圖分類、回覆編排、歷史儲存

【團隊共用】
・auth 登入、home 首頁入口、core/router.dart

【Supabase 雲端】
・orders、order_items（物資訂購）
・assistant_chat_sessions（對話歷史）
・products 等商品資料

資料流：首頁入口 → 柑仔店／小幫手 → Repository → Supabase

4.2 目錄結構（本人負責部分）

lib/features/shop/
　data/shop_products_repository.dart … 商品
　data/shop_orders_repository.dart … 需求單 CRUD
　data/product_images.json … 商品圖
　presentation/shop_page.dart … 柑仔店主頁
　presentation/shop_elder_orders_page.dart … 我的需求

lib/features/assistant/
　data/assistant_intent.dart … 意圖分類
　data/assistant_reply_orchestrator.dart … 統一回覆
　data/assistant_snapshot_loader.dart … 含代購訂單
　data/assistant_history_repository.dart … 對話歷史
　presentation/assistant_page.dart … 小幫手頁
　presentation/assistant_voice_provider.dart … 語音

supabase/
　orders_schema.sql
　assistant_chat_schema.sql
　orders_volunteer_update_rls.sql

4.3 資料庫設計（與本人模組相關）

【orders 訂單主檔】
id（主鍵）、user_id（長輩）、status、total_amount、created_at

【order_items 訂單明細】
id、order_id、product_id、product_name、quantity、unit_price

【assistant_chat_sessions 對話場次】
id、user_id、title、messages（JSON）、updated_at

關係：一筆 orders 有多筆 order_items；一使用者有多筆 assistant_chat_sessions。

RLS 原則（摘要）：
長輩僅能讀寫自己的 orders；志工依團隊 policy 可讀取並更新訂單；對話紀錄僅本人可見。

4.4 訂單狀態流程

開始 → pending（長輩送出）
pending → processing（志工處理中）
processing → completed（完成）或 cancelled（取消）
completed／cancelled → 結束

4.5 小幫手回覆流程（統一輕鬆語氣）

使用者輸入
　↓
AssistantIntent.classify（分類）
　├─ casual（閒聊）→ 陪聊模板（可選 Ollama）
　├─ systemData（查代購）→ ReplyService 查 Snapshot.recentOrders
　└─ appGuide（帶路）→ ReplyService 柑仔店教學＋導航按鈕
查詢／帶路類回覆 → AssistantTone.warmify（輕鬆語氣包裝）
　↓
顯示回覆，可寫入 assistant_chat_sessions

與物資訂購的關聯：
AssistantSnapshotLoader 呼叫 ShopOrdersRepository.listOrdersWithItemsForElder()，
小幫手回答「代購到哪」時以資料庫為準，不捏造進度。

第五章 系統實作
════════════════════════════════════════

5.1 物資訂購模組

5.1.1 商品展示
・shopProductsProvider 載入商品；圖片由 product_images.json 或 Supabase。
・卡片顯示品名、數量調整、加入需求清單。
・可開啟全聯 pxbox 搜尋結果頁作價格參考。

5.1.2 送出需求單
・ShopOrdersRepository.createOrderWithItems()：
　建立 orders（status = pending，total_amount 為明細加總）
　批次寫入 order_items（product_id、product_name、quantity 等）
・成功後清空購物車並提示長輩。

5.1.3 我的需求
・路由 /shop/orders，ShopElderOrdersPage 列出個人訂單。
・顯示狀態中文、時間、品項摘要；可下拉重新整理。

5.1.4 志工端資料層（介接）
・listOrdersWithItemsForVolunteer()：含長輩姓名、電話。
・updateOrderStatus()：更新 processing、completed 等（志工 UI 由隊友實作）。

5.2 小幫手模組

5.2.1 意圖分類（assistant_intent.dart）

casual …… 例：「最近好嗎」→ 陪聊，不查資料庫
systemData … 例：「代購到哪」→ 查 recentOrders
appGuide …… 例：「怎麼用柑仔店」→ 教學＋/shop 按鈕

閒聊優先規則：避免「最近好嗎」被誤判為查資料。

5.2.2 回覆編排（assistant_reply_orchestrator.dart）

（1）classify(question)
（2）閒聊 → AssistantCasualChatService
（3）否則 → AssistantReplyService → AssistantTone.warmify()
（4）回傳文字與導航按鈕（如 /shop、/shop/orders）

5.2.3 語音與字幕
・speech_to_text 即時辨識
・AssistantVoiceLivePanel：麥克風、音量、大字幕、確認／取消
・結束後可填入輸入框並送出

5.2.4 對話歷史
・需執行 assistant_chat_schema.sql 建表
・Supabase 優先，失敗時 SharedPreferences 備援
・側邊欄：進行中、今天、昨天、依日期分組；可刪除、繼續對話
・離開頁面僅同步，不強制結案；「新對話」才換場次存檔

5.3 與首頁的介接（團隊）
・首頁小幫手捷徑 → /assistant
・底部導覽「柑仔店」→ ShopPage
・小幫手帶路可使用 /home?tab=1 切換柑仔店分頁

7.1 結論

本人於專題中完成物資需求單訂購流程與智慧小幫手兩大模組，使長輩能以簡化步驟送出代購需求、查詢真實進度，並以語音與輕鬆對話降低使用門檻。資料存於 Supabase，與志工端透過共用 orders 表協作；小幫手以意圖分類與規則引擎達成不付費 API 仍可查真實訂單之目標。

7.2 困難與解決

order_status 與程式字串不一致
　→ order_status_enum_fix.sql、常數 initialOrderStatus

小幫手離開頁面即結案對話
　→ 改為僅新對話／換帳號才存檔結案

閒聊被誤判為查資料
　→ AssistantIntent 閒聊優先規則

Web 語音需麥克風權限
　→ Chrome localhost 測試說明

7.3 未來改進（本人模組）

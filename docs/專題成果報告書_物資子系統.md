# 第三章、專題理論部份

## 3.3 Hybrid NLU 驅動之高齡友善智慧物資代購協作系統

本子系統針對偏鄉長輩日常採買困難，建立以語音互動、志工協作與即時同步為核心的物資代購平台，理論基礎涵蓋需求分析、對話系統 NLU、離線優先架構與事件驅動協作。

### 3.3.1 偏鄉高齡者物資採買需求與數位落差

隨著台灣邁入高齡社會，偏鄉社區的長輩在日常採買上面臨的困難，已不僅是「買不到東西」，而是「無法順利把需求說清楚、傳達出去」。以本專題實踐場域苗栗縣明德社區為例，當地青年外移、公車班次有限，長輩若要購買民生用品，多半仍仰賴家人載送、鄰里口頭請託，或由社區志工代為記錄後統一採買。此種人際互助模式雖具在地溫度，卻在需求變多、志工人力有限時，逐漸暴露出結構性限制。

田野訪談與據點觀察顯示，現行做法主要存在三項痛點。第一，**需求紀錄容易失真**：紙條、電話或口頭轉述難以保留完整資訊，品項名稱、數量、品牌或規格常在傳遞過程中遺漏，導致志工到賣場後仍需反覆確認。第二，**一般電商並非為長輩設計**：現有購物平台強調搜尋、比價、優惠與線上付款，畫面資訊密集、操作步驟繁瑣，對視力衰退、不熟悉智慧型手機的長輩而言學習成本過高，也無法直接連結社區志工的代購服務。第三，**志工缺乏可用的數位協作工具**：一位志工往往需同時回應多位長輩的採買請託，卻缺少能即時彙整「今日要買什麼」、標記「哪一項已買到」的系統，仍高度依賴人工抄寫與記憶，行政負擔重且易發生疏漏。

上述問題的本質，可從數位落差與科技接受兩個理論面向理解。Van Dijk（2020）指出，數位落差不僅是「有沒有設備或網路」的問題，更涉及使用動機、操作技能與能否從數位服務中獲得實質幫助；偏鄉長輩常同時卡在多個層次。Davis（1989）的科技接受模型（TAM）則進一步說明，使用者是否願意採用新系統，很大程度取決於「好不好用」——對高齡族群而言，若操作步驟過多、回饋不夠清楚，即使功能再多也難以真正進入日常生活。

基於上述分析，本專題物資子系統不以複製商業電商為目標，而是設計一套符合明德社區實際服務情境的**高齡友善代購協作平台**：讓長輩能以語音或簡單操作記錄需求，在三步內完成送出；讓志工能即時收到清單、彙整採買品項並回報進度；讓據點得以掌握服務狀況，逐步弭平偏鄉長輩在物資採買上的數位落差。

### 3.3.2 服務設計與利害關係人分析

本專題採服務設計（Service Design）方法，以「長輩記需求 → 志工採買履行 → 據點營運檢視」為主軸繪製服務藍圖。物資子系統**僅區分長輩（`elder`）與志工（`volunteer`）兩種登入角色**；據點統計、滯留單與資產檢視整合於志工主控台「數據總覽」分頁，符合社區關懷據點「一人多工」之實務，避免額外管理端登入造成使用複雜度。

| 利害關係人 | 核心需求 | 主要痛點 | 系統對應功能 |
|-----------|---------|---------|------------|
| 長輩 | 語音記需求、查參考價、追蹤代購進度 | 不會用電商 App、口語模糊難一次說清 | 柑仔店／需求單、智慧小幫手、訂單 Realtime |
| 志工 | 彙整採買清單、更新履行狀態、掌握據點概況 | 紙條散亂、多長輩難整合 | 今日採買清單 RPC、品項履行、數據總覽 Tab |

高齡友善設計原則貫穿本子系統：字體 ≥ 24pt、按鈕觸控區 ≥ 48dp、核心任務（語音記需求 → 確認品牌 → 送出志工）控制在三步以內，並提供 TTS 語音播報輔助閱讀。

### 3.3.3 自然語言理解與 Hybrid NLU 理論

#### （1）槽位填充對話系統

智慧小幫手本質為**槽位填充對話系統**（Slot-Filling Dialogue System），包含兩個核心模組：

- **意圖識別（Intent Recognition）**：判斷使用者目的，如「記錄需求」「查詢價格」「取消需求」。
- **槽位填充（Slot Filling）**：從語句萃取參數，如商品名稱、數量、品牌偏好。

以「我要衛生紙兩包」為例，意圖為「記錄需求」，槽位為 `{商品: 衛生紙, 數量: 2}`。偏鄉長輩語句常不完整（如只說「再來兩包」），系統需透過**澄清反問（Clarification Prompting）**補齊槽位，而非直接失敗。

#### （2）規則式 NLU 與混合式架構

對話系統的 NLU 實作可分為規則式、機器學習式與大語言模型式三類。規則式方法延遲低、可解釋、可離線運行；LLM 理解力強，卻有成本高、延遲大、斷網不可用與輸出不可控等風險。

本專題採 **Hybrid NLU（混合式自然語言理解）** 策略：

```
語音/文字輸入
    ↓
L1 精確關鍵字 → L2 正則句型 → L3 同義詞映射（端側規則，<50ms）
    ↓
商品語意對齊（product_items / product_aliases）
    ↓
信心評估（confidence）
    ├─ ≥ 0.75 且槽位完整 → 直接執行業務動作
    ├─ 槽位不足 → clarification_sessions 多輪澄清
    └─ 信心不足 → 可選呼叫 Edge parse_shop_nlu（JSON Schema 約束）
```

**代購核心路徑不依賴雲端 LLM**，確保偏鄉斷網時仍可分類與暫存；一般閒聊經 Edge `assistant_casual_chat`（Gemini）或本地模板處理，與代購路徑分離。規則引擎可抽象為**確定型有限自動機（DFA）**，便於口試展示與除錯。

#### （3）語音互動與高齡 VAD 考量

語音互動鏈路為：**VAD → ASR → NLU → 業務動作 → TTS 播報**。高齡使用者語速慢、停頓多，本子系統採：

1. **聽寫模式**（`ListenMode.dictation`）搭配即時字幕；
2. **按住說話（Push-to-Talk）** 與延長 `pauseFor`（3.5 秒）。

TTS 播報（`flutter_tts`，zh-TW、語速 0.45）降低閱讀負擔；收音前停止 TTS，避免回音干擾辨識。

### 3.3.4 離線優先與即時協作理論

#### （1）Offline-First 架構

偏鄉網路品質不穩，系統採 **Offline-First（離線優先）** 設計：離線可完成意圖分類、槽位填充與口語商品解析；連線後再同步至伺服器。本系統以 Hive 實作本機佇列，每筆待送資料附 **UUID v4 冪等鍵（`clientRequestId`）**，避免恢復連線後重複建單。

#### （2）即時協作與事件驅動同步

志工代購屬多方協作場景，本系統改採 **事件驅動架構**：Supabase Realtime 訂閱 PostgreSQL 資料列變更，透過 WebSocket 推送至客戶端。本地 Wi-Fi 環境實測延遲約 300–800ms。

#### （3）雙軌狀態模型

| 層級 | 欄位 | 用途 |
|------|------|------|
| 訂單級 | `orders.status` | pending / processing / completed（志工主列表） |
| 品項級 | `demand_record_items.fulfillment_status` | 已購買、已發放、替代等（今日採買／履行） |

訂單級便於掌握處理進度；品項級支援同一趟採買、多品項分別履行。兩軌並行為過渡設計，後續可統一為單一履行時間軸。

### 3.3.5 相關技術與現有方案比較

| 構面 | 主流電商（蝦皮、momo） | 本專題物資子系統 |
|------|----------------------|-----------------|
| 目標用戶 | 一般消費者 | 偏鄉高齡長輩 |
| 輸入方式 | 搜尋、點選、結帳流程 | 語音為主、三步送出 |
| 志工協作 | 無 | 今日採買清單、Realtime、履行追蹤 |
| 離線能力 | 幾乎無 | 規則 NLU + Hive 佇列 |
| 付款 | 線上支付為核心 | 本期不納入（社區代購現場結算） |
| 推播整合 | 商業推播 | 長輩送單／志工接單 FCM（Android） |

現有方案無法整合社區志工服務體系，亦未針對高齡語音習慣優化。本專題填補「偏鄉高齡者 × 志工協作 × 語音代購」之整合性缺口。

### 3.3.6 研究問題

本專題針對物資子系統提出以下可驗證問題：

- **RQ-Shop-1**：三層規則式意圖分類器，能否在偏鄉長輩口語語料上達到 ≥ 90% 準確率？

上述問題於 **4.3 實作與測試** 中驗證。

# 第四章、專題實作部份

## 4.3 Hybrid NLU 驅動之高齡友善智慧物資代購協作系統

物資子系統採 Flutter 前端 + Supabase 後端，實作長輩語音記需求、志工採買履行與據點數據檢視之完整閉環。

### 4.3.1 系統架構與技術選型

| 技術項目 | 選型 | 說明 |
|---------|------|------|
| 前端框架 | Flutter (Dart) | Material 3 高齡友善元件 |
| 狀態管理 | Riverpod | 非同步狀態、Realtime 訂閱 |
| 語音輸入 | speech_to_text | 中文聽寫，`pauseFor=3.5s` |
| 語音輸出 | flutter_tts | zh-TW 播報，與收音互斥 |
| 離線儲存 | Hive + connectivity_plus | 本機佇列、連線監聽 |
| NLU 編排 | HybridNluOrchestrator | 規則優先，信心 < 0.75 可選 Edge |
| 後端 | Supabase BaaS | RLS、Realtime、Edge Functions |
| 推播 | firebase_messaging + send_shop_push | Android FCM v1 |

**整體資料流：**

```
長輩（語音/文字）
  → shop_utterance_handler / HybridNluOrchestrator
  → demand_records（草稿）或 orders（正式單）
  → Supabase Realtime
  → 志工端列表 / 今日採買清單 / FCM 推播
```

### 4.3.2 前端實作

#### （1）長輩端

柑仔店 `/shop` 為**唯一主入口**（`/shop/demand-input` 自動導向 `/shop`）。

| 路由 | 功能 | 主要程式 |
|------|------|---------|
| `/shop` | 需求單、語音、品牌確認、送出志工 | `shop_page.dart`、`shop_primary_demand_section.dart` |
| `/shop` | 按住說話、多品項語音解析 | `shop_manual_voice_section.dart`、`shop_manual_voice_parser.dart` |
| `/shop/prices` | 全聯價格參考 | `shop_price_page.dart` |
| `/shop/orders` | 我的需求單、配送進度 | `shop_elder_orders_page.dart` |
| `/shop/orders/:id` | 訂單詳情、配送時間軸 | `shop_order_detail_page.dart` |
| `/assistant` | 智慧小幫手 Hybrid NLU | `assistant_page.dart`、`hybrid_nlu_orchestrator.dart` |
| `/assistant/history` | 對話歷史 | `assistant_chat_history_page.dart` |

**Demo 主線：** 語音「我要衛生紙兩包」→ 選品牌 → **送出給志工** → 「我的需求單」追蹤進度。

#### （2）志工端

| 路由 | 功能 | 主要程式 |
|------|------|---------|
| `/volunteer/shop-orders` | 訂單列表、接單、今日採買清單 | `volunteer_shop_orders_page.dart`、`volunteer_daily_shopping_list_panel.dart` |
| 同上 | 品項履行（已購／已發／替代） | `volunteer_item_fulfillment_sheet.dart` |
| `/volunteer-dashboard?tab=3` | 數據總覽 | `volunteer_hub_analytics_tab.dart` |

志工 **接單（代購）** 後訂單由 `pending` 轉為 `processing`；長輩端經 Realtime 即時更新。

### 4.3.3 Hybrid NLU 實作

`assistant_shop_intent_classifier.dart` 實作三層瀑布式匹配：

| 層級 | 策略 | 範例 |
|------|------|------|
| L1 | 精確關鍵字 | 「買」「多少錢」「不要了」 |
| L2 | 正則句型 | 「我要買[數量][商品]」 |
| L3 | 同義詞庫 | 「購買」「採買」→ 記錄需求 |

`HybridNluOrchestrator` 編排流程：規則分類 → 商品語意對齊 → 信心評估 → 澄清或 Edge `parse_shop_nlu` 補強。業務動作由 `assistant_shop_action_service.dart` 執行；統一入口 `shop_utterance_handler.dart` 連結柑仔店與小幫手。分類實測 **< 50ms**；閒聊經 `assistant_casual_chat`，與代購路徑分離。

### 4.3.4 後端與資料庫實作

**核心資料表：**

| 資料表 | 用途 |
|--------|------|
| `demand_records` / `demand_record_items` | 需求草稿、品項履行狀態 |
| `orders` / `order_items` | 正式訂單 |
| `product_items` / `product_aliases` | 商品語意庫 |
| `price_references` | 全聯參考價 |
| `clarification_sessions` | 多輪澄清 |
| `device_tokens` | FCM 推播（Android） |

**遷移順序**（見 `supabase/DEPLOY.md`）：基礎 schema → `20260601`～`20260607`（含 Realtime）→ `price_references_demo_seed.sql`。

**Edge Functions：** `parse_shop_nlu`（NLU 補強）、`send_shop_push`（FCM）、`assistant_casual_chat`（閒聊）。

**RPC：** `get_daily_shopping_list` 聚合今日採買品項；Realtime 訂閱見 `shop_orders_realtime_provider.dart`。

### 4.3.5 離線佇列與推播實作

`OfflineQueue`（`offline_queue.dart`）：

- 斷網時序列化至 Hive，附 `clientRequestId` 冪等鍵；
- 恢復連線後 `flush()` 重傳草稿，最多重試 5 次；
- 正式訂單須柑仔店按「送出給志工」才建立 `orders` 並觸發 FCM。

Android 已接入 `firebase_messaging`；iOS 尚未接入 APNs。

## 2.2 技術

**物資子系統：**

| 層級 | 技術 | 說明 |
| --- | --- | --- |
| 前端框架 | Flutter 3.x（Dart） | 跨平台 App、Material 3 高齡友善 UI |
| 前端狀態 | Riverpod | 非同步狀態、Realtime 訂閱、編譯期安全 |
| 前端路由 | GoRouter + RoleGuard | 長輩／志工角色分流 |
| 後端平台 | Supabase BaaS | Auth、PostgreSQL、Realtime、Storage |
| 無伺服器函式 | Supabase Edge Functions（Deno/TypeScript） | NLU 補強、FCM 推播、閒聊 |
| 身分驗證 | Supabase Auth（Email + OTP） | 單一帳號；`profiles.role` 區分 elder／volunteer |
| 權限控管 | PostgreSQL RLS（Row Level Security） | 長輩僅存取自身訂單；志工可讀寫協作資料 |
| 語音輸入 | speech_to_text | 中文聽寫；`pauseFor=3.5s` 適應高齡語速 |
| 語音輸出 | flutter_tts（zh-TW） | 小幫手回覆播報；與收音互斥 |
| NLU 引擎 | Hybrid NLU（端側規則 + 可選 Edge） | L1 關鍵字 → L2 正則 → L3 同義詞；信心 < 0.75 可呼叫 `parse_shop_nlu` |
| 離線佇列 | Hive + connectivity_plus | 斷網暫存需求草稿；`clientRequestId` 冪等重傳 |
| 即時同步 | Supabase Realtime（WebSocket） | 訂閱 `orders`、`demand_records` 變更 |
| 推播 | firebase_messaging + FCM HTTP v1 | Android 志工／長輩鎖屏通知（Edge `send_shop_push`） |
| 閒聊（非代購路徑） | Edge `assistant_casual_chat`（Gemini） | 與代購核心路徑分離；離線降級本地模板 |
| 圖表（志工統計） | fl_chart | 數據總覽 Tab 訂單狀態統計 |

## 2.3 後端模組設計

**物資子系統：**

| 模組／機制名稱 | 核心功能與設計說明 |
| --- | --- |
| Auth（身分驗證） | 支援 Email 登入／註冊；註冊時可勾選「社區志工」；系統依據使用者角色自動區分長輩與志工 |
| PostgreSQL（資料儲存） | 結構化儲存物資子系統核心資料表，包含代購訂單、需求草稿、標準商品主檔、通路參考價、對話紀錄、社區據點與推播憑證登記等 |
| Realtime（即時同步） | 透過即時連線建立資料串流；長輩送出需求或志工更新訂單進度時，雙方 App 畫面秒級同步，無需手動重新整理 |
| Edge Function（邊緣運算） | 負責三大雲端任務：語意補強（口語不清楚時再次解析）、推播通知（接單與送達提醒）、閒聊回覆（與代購路徑分開） |
| RLS（資料列權限控管） | 長輩僅能存取自己的需求草稿與訂單；志工僅可讀寫協作範圍內的訂單與品項履行資料；小幫手對話紀錄僅本人可見 |
| 本機離線暫存 | 結合手機本機儲存與連線偵測；斷網時先將需求存於手機，恢復連線後自動上傳草稿，避免重複建立相同品項 |
| 端側語意理解 | 手機端先行判斷「要買什麼、買幾個」；語句不完整時反問補充；一句話說多項商品可自動拆分 |
| 今日採買彙整 | 資料庫後端依據點與日期，即時將多位長輩需求合併成「今日採買清單」，取代志工紙筆抄寫 |
| 個人化推薦 | 串接購買與推薦紀錄，依長輩常買類別、平價偏好與歷史購買頻率，顯示三張推薦卡片協助快速選品 |

## 2.4 資料庫設計

**物資子系統：**

#### 代購訂單與需求

| 資料表 | 中文名稱 | 核心用途 |
| --- | --- | --- |
| orders | 代購訂單主檔 | 儲存長輩正式送出的代購單，含處理狀態、所屬據點、負責志工、送達時間與備註 |
| order_items | 訂單明細 | 記錄每一張訂單內的商品名稱、數量與參考單價 |
| demand_records | 需求草稿主檔 | 儲存長輩尚未送出或已轉正式單前的需求單，可對應到最終訂單 |
| demand_record_items | 需求明細 | 記錄各品項名稱、數量，以及志工是否已購買、已送達或改買替代品 |
| order_delivery_events | 配送進度紀錄 | 記錄訂單從建立、接單、採買到送達各階段時間點，供長輩查看進度 |
| demand_fulfillment_events | 品項履行紀錄 | 追蹤單一品項狀態變更歷程，便於志工與據點事後查核 |

#### 商品、價格與推薦

| 資料表 | 中文名稱 | 核心用途 |
| --- | --- | --- |
| product_categories | 商品類別 | 定義衛生紙、雞蛋等採買分類，供柑仔店選品與推薦使用 |
| product_brands | 品牌與規格 | 儲存各品牌名稱、包裝規格與參考價格 |
| product_items | 標準商品主檔 | 將類別、品牌、規格整合為系統可辨識的標準品項 |
| product_aliases | 口語同義詞 | 將長輩常說的別名（如「衛生紙」）對應到標準商品，協助語音理解 |
| price_references | 參考價格 | 儲存全聯等通路參考價，供小幫手查價與價格頁顯示 |
| purchase_events | 購買紀錄 | 記錄長輩過去購買行為，作為個人化推薦依據 |
| recommendation_logs | 推薦紀錄 | 記錄系統曾向長輩推薦哪些商品，供日後分析與優化 |

#### 智慧小幫手與據點協作

| 資料表 | 中文名稱 | 核心用途 |
| --- | --- | --- |
| assistant_chat_sessions | 對話紀錄主檔 | 儲存長輩與小幫手每一場對話的內容與時間 |
| chat_histories | 對話歷史檢視 | 供 App 讀取對話列表的檢視表，底層資料來自對話紀錄主檔 |
| clarification_sessions | 澄清對話狀態 | 當長輩語句不完整時，暫存反問與補充資訊直至需求釐清 |
| location_points | 社區據點 | 儲存明德里活動中心等據點名稱、地址與聯絡方式 |
| location_assets | 據點物資清單 | 記錄據點現有物資數量，供志工數據總覽檢視 |
| daily_shopping_lists | 今日採買清單 | 可選擇性儲存當日彙整後的採買清單（主流程亦支援即時彙整） |
| device_tokens | 推播裝置登記 | 記錄使用者手機推播識別，用於發送接單與送達通知 |
| push_notifications_log | 推播發送紀錄 | 記錄每次通知是否成功送達，便於除錯與追蹤 |

**權限設計摘要**：長輩僅能讀寫自己的需求單與訂單；志工可讀寫協作範圍內的訂單、品項履行與據點資料；對話紀錄僅本人可存取，避免隱私外洩。

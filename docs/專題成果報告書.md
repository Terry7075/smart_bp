# 第一章、緒論前言

隨著台灣正式邁入高齡社會，偏鄉地區的高齡化現象尤為顯著。以苗栗縣明德社區為例，當地青年人口外流，社區內以長者居多。在這樣的背景下，社區治理與長者照護面臨了雙重挑戰：一方面，村里辦公室與社區發展協會的行政與志工資料缺乏有效管理；另一方面，長者在健康維護、外出交通以及日常採買上，皆因資源匱乏與數位落差而面臨重重困難。因此，本專題以明德社區為實踐場域，期望透過資訊科技的介入，打造一套切合在地需求的數位整合平台。

## 1.1 研究背景

經深入社區探訪與需求盤點，我們歸納出現行偏鄉高齡社區面臨的四大核心痛點：

**（一）社區資料管理混亂（治理端）**  
村里辦公室的公文書、志工排班表及長者照護紀錄多散落於實體紙本或個人裝置中，缺乏統一的儲存空間、嚴謹的權限控管與高效的檢索機制，導致交接困難與資安隱患。

**（二）用藥安全與健康風險（健康端）**  
偏鄉長者常患有多種慢性病，藥袋資訊繁雜且字體細小，導致長者經常忘記服藥或吃錯藥；同時，慢性病連續處方箋的代領流程繁瑣，缺乏數位化追蹤。

**（三）公共交通資源匱乏（出行端）**  
偏鄉大眾運輸班次稀少，長者無論是前往市區就醫或採買皆極度不便。現有的計程車叫車軟體（如 Uber、LINE GO）介面過於複雜，且費用高昂，不符合偏鄉長者需求。

**（四）數位落差與採買不便（生活端）**  
長者難以適應現代主流電商平台複雜的結帳與搜尋流程，日常民生物資採買常需仰賴鄰里口頭請託，缺乏一個直覺、無障礙且整合語音輔助的數位媒合管道。

上述痛點並非彼此孤立，而是形成連鎖效應：長輩若無法安全服藥，可能增加就醫頻率；若交通不便，則難以領取處方箋；若採買依賴口頭請託，志工亦難以有效彙整與追蹤。現有商業 App 多以都市年輕用戶為設計對象，缺乏針對偏鄉高齡者「整合性、在地化、志工協作」的解決方案。

## 1.2 研究目的

針對上述痛點，本專題提出「明德 e 達人」整合服務平台，目標是弭平數位落差，建立一個以「智慧里政大腦」為後盾、「高齡友善服務」為前端的完整生態系：

1. **建構里政資料大腦**  
   開發基於 NAS 的檔案管理控制台，利用 AI 語意搜尋與 RBAC 權限控管，實現社區資料的安全儲存與智慧檢索。

2. **建立智慧用藥防護網**  
   結合 AI 視覺辨識（OCR），將實體藥袋數位化，並提供看圖認藥、定時提醒與處方箋代領追蹤功能，確保長者健康。

3. **打造在地互助出行平台**  
   開發極簡化的智行共乘系統，媒合社區內有車的村民與有出行需求的長者，解決就醫與採買的交通障礙。

4. **實現無障礙數位生活**  
   建置符合長者直覺的物資需求系統，並導入「智慧小幫手」，以語音與口語化互動取代傳統鍵盤輸入，協助長者輕鬆完成代購與資訊查詢。

## 1.3 研究範圍與方法

本專題研究範圍涵蓋四個子系統之需求分析、系統設計、程式實作與功能測試，服務場域聚焦苗栗縣客家聚落社區關懷據點，目標用戶為 65 歲以上長輩、社區志工、檔案管理人員與共乘司機。

研究方法採「服務設計」與「敏捷開發」並行：

- **服務設計**：透過田野訪談、利害關係人分析與高齡友善設計原則（大字體、高對比、三步內完成核心任務），確保功能貼合實際情境。
- **技術實作**：前端以 Flutter 跨平台框架統一長輩端體驗；後端以 Supabase BaaS 為核心，並依子系統特性整合 NestJS、PostgreSQL、Redis 及多項 AI 服務。
- **驗證方式**：以功能測試、單元測試與 Demo 情境驗收，確認各子系統核心流程可正常運作。

## 1.4 團隊分工

本專題由四位資工系大三學生組成，採「統一平台、分項實作、共享基礎」之協作模式。

| 子系統 | 核心技術 | 主要功能 |
|--------|---------|---------|
| 智慧用藥與社區學習 | Flutter、Supabase、ML Kit、Gemini | 藥袋 AI 辨識、慢箋代領、看圖認藥、本機提醒、社區學習 CMS |
| NAS 檔案管理控制台 | React、NestJS、PostgreSQL、COCO-SSD、Sentence-BERT | 檔案瀏覽、RBAC 權限、AI 語意標籤、語意搜尋、審計記錄 |
| 智慧物資代購協作 | Flutter、Supabase、Hybrid NLU、Hive 離線佇列 | 柑仔店需求單、智慧小幫手、今日採買清單、志工數據總覽 |
| 智行共乘交通媒合 | Flutter、Supabase、GPS 定位 | 共乘媒合、固定接送、即時定位、公車資訊 |

四子系統共享 Supabase 身份驗證、PostgreSQL 資料庫、Material 3 高齡友善 UI 規範（字體 ≥ 24pt、按鈕 ≥ 48dp），以降低長輩跨功能學習成本。

# 第二章、系統架構及介紹

## 2.1 系統總體架構

「明德 e 達人」採四層式整合架構，由上至下分別為：**使用者層**、**前端呈現層**、**後端服務層**與**資料／AI 層**。

```
┌─────────────────────────────────────────────────────────────────┐
│                        使用者層                                  │
│   長輩（65+）  │  社區志工  │  檔案管理者  │  共乘司機           │
└────────┬──────────────┬──────────────┬──────────────┬──────────┘
         │              │              │              │
┌────────▼──────────────▼──────────────▼──────────────▼──────────┐
│                     前端呈現層                                   │
│  Flutter App（用藥／物資／共乘）  │  React Web（檔案管理）        │
│  Material 3 高齡友善 UI · GoRouter 路由 · RoleGuard 權限        │
└────────┬────────────────────────────────────────────────────────┘
         │
┌────────▼────────────────────────────────────────────────────────┐
│                     後端服務層                                   │
│  Supabase BaaS：Auth · PostgreSQL · Storage · Realtime          │
│  Edge Functions：藥袋辨識 · NLU 補強 · FCM 推播                  │
│  NestJS API：檔案 RBAC · 語意搜尋 · 審計記錄                     │
└────────┬────────────────────────────────────────────────────────┘
         │
┌────────▼────────────────────────────────────────────────────────┐
│                   資料／AI 能力層                                │
│  ML Kit OCR │ Gemini 1.5 Flash │ COCO-SSD │ Sentence-BERT        │
│  Hybrid NLU（規則式意圖分類）│ Hive 離線佇列 │ GPS 定位           │
└─────────────────────────────────────────────────────────────────┘
```

**架構設計原則：**

1. **單一登入**：長輩與志工以一組帳號通行各子系統，RLS（Row Level Security）確保資料列級權限。
2. **混合式 AI**：端側 OCR／規則 NLU 保障離線可用；雲端 LLM 僅於高價值場景（藥袋語意萃取、閒聊）補強。
3. **即時協作**：Supabase Realtime 取代短輪詢，長輩送出需求後志工端於約 300–800ms 內收到更新。
4. **高齡友善一致**：森林綠（#4A6741）為長輩端主色、岩藍（#3D5A80）為志工／管理端主色。

## 2.2 智慧用藥與社區學習子系統

本子系統解決偏鄉長者「看不懂藥袋、忘記服藥、慢箋代領繁瑣」等問題，並延伸社區學習內容推播。

**核心功能：**

| 功能 | 說明 |
|------|------|
| 藥袋 AI 掃描 | 拍照後以 ML Kit 離線 OCR + Gemini 語意萃取，自動建立藥單 |
| 看圖認藥 | 對照藥典圖片，降低化學藥名閱讀障礙 |
| 吃藥提醒與打卡 | 本機通知排程，追蹤用藥依從性 |
| 慢箋志工代領 | 無領藥日處方箋自動建立志工任務，支援批次代領與健保卡鎖 |
| 社區學習 CMS | 志工上傳防詐騙、健康衛教、客語文化內容，Realtime 推播至長輩端 |

**技術架構：** Flutter 前端 → Supabase REST／Realtime → Edge Function `process_prescription_vision` → Gemini API → PostgreSQL（`prescriptions`、`medication_logs`、`volunteer_tasks`）。

## 2.3 NAS 檔案管理控制台

本子系統針對社區關懷據點累積之計畫報告、活動照片、評鑑資料等，提供現代化檔案治理能力。

**核心功能：**

| 功能 | 說明 |
|------|------|
| 檔案瀏覽與上傳 | 樹狀目錄、拖曳上傳、批次操作 |
| RBAC 權限治理 | 系統管理員、檔案管理員、一般使用者、訪客四級角色 |
| AI 語意標籤 | COCO-SSD 瀏覽器端物件偵測 + Sentence-BERT 語意向量 |
| 語意搜尋 | 以自然語言查詢，突破關鍵字匹配限制 |
| 審計記錄 | 完整操作軌跡，支援稽核與追溯 |

**技術架構：** React 18 + TypeScript 前端 → NestJS 模組化後端 → PostgreSQL + Redis 快取 → 本機磁碟／NAS 掛載儲存。

## 2.4 智慧物資代購協作子系統

本子系統以「Hybrid NLU 驅動之高齡友善智慧物資代購協作」為定位，**僅區分長輩與志工兩種登入角色**，志工主控台第四分頁「數據總覽」整合據點營運統計，不另設獨立管理 App。

**角色與流程：**

```
長輩 ──語音/文字需求──► Hybrid NLU ──► demand_records / orders ──► 志工
                                                              │
                    志工 ◄── Realtime ── 今日採買清單 / 品項履行 ◄──┘
```

**核心功能：**

| 功能 | 說明 |
|------|------|
| 柑仔店／需求單 | 長輩以語音或文字記錄採買需求，三步內完成送出 |
| 智慧小幫手 | L1 關鍵字 → L2 正則句型 → L3 同義詞三層意圖分類；低信心時可選 Edge `parse_shop_nlu` 補強 |
| 離線優先 | `OfflineQueue`（Hive）斷網暫存，恢復連線後冪等重傳草稿 |
| 今日採買清單 | RPC `get_daily_shopping_list` 聚合志工採買品項 |
| 志工數據總覽 | 訂單統計、滯留單、熱門品項 Top5、據點資產檢視 |

**NLU 設計重點：** 代購核心路徑不依賴雲端大語言模型，確保偏鄉斷網時仍可分類與暫存；閒聊經 Edge `assistant_casual_chat`（Gemini）或本地模板，與代購路徑分離。

## 2.5 智行共乘交通媒合子系統

本子系統針對偏鄉公車班次稀少、商業叫車平台不符社區需求之現況，提供「公告制共乘」媒合服務。

**核心功能：**

| 功能 | 說明 |
|------|------|
| 地點快速選擇 | 預設苗栗醫院、市場等常用地點，一鍵發出需求 |
| 自動媒合 | 依時間、地點、剩餘座位三維度篩選司機 |
| 固定接送 | 支援每週固定回診行程，減少重複操作 |
| 即時定位 | GPS 追蹤司機位置，降低長輩等待焦慮 |
| 簡化車資 | 5 公里內 20 元、5–10 公里 50 元、10 公里以上 100 元 |

**技術架構：** Flutter 前端 → Supabase（行程資料、Realtime 通知）→ 裝置 GPS 定位服務。

# 第三章、專題實作部份

## 3.1 智慧用藥子系統實作

### 3.1.1 前端實作

前端以 Flutter 3.x 開發，狀態管理採 Riverpod，路由採 GoRouter 並以 `RoleGuard` 區分長輩與志工分支。

| 路由 | 功能 | 目標角色 |
|------|------|---------|
| `/health-scan` | 藥袋拍照、上傳、AI 辨識進度 | 長輩 |
| `/health` | 藥單列表、Realtime 狀態監聽 | 長輩 |
| `/medication-checkin` | 看圖認藥、吃藥打卡 | 長輩 |
| `/community-learning` | 社區學習內容瀏覽 | 長輩 |
| `/volunteer-dashboard` | 藥單協助、批次代領、內容 CMS | 志工 |

本機通知採 `flutter_local_notifications`，APP 冷啟動時重建排程，確保提醒不依賴網路。

### 3.1.2 AI 辨識引擎

採混合式架構：

1. **第一階段（端側）**：ML Kit 離線 OCR 擷取藥袋文字，無網路時可暫存。
2. **第二階段（雲端）**：影像上傳 Storage 後，Edge Function 呼叫 Gemini 1.5 Flash，以 JSON Schema 輸出醫療機構、領藥日期、藥物陣列等結構化資料。
3. **降級機制**：Gemini 429 限流時指數退避重試；5xx 錯誤時降級為純 OCR 文字模式。

### 3.1.3 慢箋代領流程

當辨識結果 `dispensed_date` 為 null（慢箋）時，系統自動建立 `volunteer_tasks`。志工端支援同一藥局、領藥日期相差 10 天內的批次合併代領，並以「健保卡鎖」防止未收卡即標記領藥中。任務完成後由 Database Trigger 同步更新藥單狀態，並透過 Realtime 通知長輩端。

## 3.2 NAS 檔案管理子系統實作

### 3.2.1 前端實作

React 18 + TypeScript + Tailwind CSS，主要模組包含檔案瀏覽器（樹狀／列表雙模式）、拖曳上傳、權限管理介面、審計時間軸與語意搜尋輸入框。

### 3.2.2 後端實作

NestJS 模組化架構：

- **檔案管理模組**：CRUD、目錄管理、移動與複製。
- **權限管理模組**：RBAC 角色指派、目錄繼承式權限（Closest Ancestor Wins）。
- **AI 標籤模組**：上傳圖片時以 TensorFlow.js 載入 COCO-SSD 進行瀏覽器端物件偵測，再以 Sentence-BERT 生成 384 維語意向量寫入 PostgreSQL。
- **語意搜尋模組**：查詢文字編碼後以餘弦相似度排序，閾值 0.65，Top-5 準確率（MRR@5）約 78%。

Redis 快取層將常見路徑查詢回應時間降低約 60%。

## 3.3 物資代購協作子系統實作

### 3.3.1 前端實作

**長輩端**（路由 `/shop`）：

- `shop_primary_demand_section.dart`：文字／語音統一需求入口。
- `shop_manual_voice_section.dart`：按住說話（Push-to-Talk），`pauseFor=3.5s` 適應高齡語速。
- `shop_page.dart`：柑仔店選品、品牌確認、「送出給志工」。
- `/assistant`：智慧小幫手 Hybrid NLU 對話。

**志工端**：

- `/volunteer/shop-orders`：訂單列表、今日採買清單、品項履行。
- `/volunteer-dashboard?tab=3`：數據總覽（`volunteer_hub_analytics_tab.dart`）。

### 3.3.2 Hybrid NLU 實作

`HybridNluOrchestrator` 編排流程：

1. `AssistantShopIntentClassifier` 三層規則匹配（平均 < 50ms）。
2. 槽位不足時啟動 `clarification_sessions` 多輪澄清。
3. 信心分數 < 0.75 時可選呼叫 Edge `parse_shop_nlu`。
4. 回覆完成後以 `flutter_tts`（zh-TW、語速 0.45）自動播報，與收音互斥。

### 3.3.3 離線佇列與即時同步

`OfflineQueue`（Hive + `connectivity_plus`）：

- 斷網時將需求序列化至本機，附 UUID v4 冪等鍵 `clientRequestId`。
- 恢復連線後 `flush()` 重傳至 `demand_records` 草稿，最多重試 5 次。
- 正式訂單須長輩於柑仔店按「送出給志工」後建立，並可觸發 Edge `send_shop_push`（Android FCM）。

`shop_orders_realtime_provider.dart` 訂閱 `orders`、`demand_records` 變更，Debug 模式下 `RealtimeLatencyBanner` 量測端到端延遲約 300–800ms。

### 3.3.4 資料庫與 Edge Functions

核心資料表：`demand_records`、`demand_record_items`（含 `fulfillment_status`）、`orders`、`product_items`、`product_aliases`、`price_references`、`clarification_sessions`。

遷移順序依 `supabase/DEPLOY.md`：基礎 schema → `20260601`–`20260607`（含 Realtime publication）。查價 Demo 種子見 `price_references_demo_seed.sql`。

## 3.4 智行共乘子系統實作

### 3.4.1 長者端

- 預設常用地點一鍵選擇，支援立即／預約乘車。
- 媒合成功後顯示司機資訊、即時地圖定位與級距車資。
- 狀態推播：媒合成功、司機出發、即將抵達。

### 3.4.2 司機端與管理端

- 司機申請經管理員審核後可接單，支援可載人數設定與固定接送申請。
- 管理員可監控進行中行程、查看司機定位、必要時強制取消行程。

### 3.4.3 媒合演算法

採啟發式貪婪演算法：先篩選時間窗口內在線司機 → 檢查剩餘座位 → 以 GPS 直線距離排序，時間複雜度 O(n log n)，適用偏鄉規模（同時在線司機通常 < 20）。

|--------|---------|---------|
| 智慧用藥 | 13 項功能案例（掃描、慢箋、打卡、RLS 越權等） | 全數通過；端到端建檔約 12 秒 |

# 第四章、結論與建議

## 4.1 研究結論

本專題以苗栗明德社區為實踐場域，成功設計並實作「明德 e 達人」整合服務平台，回應偏鄉高齡者在用藥安全、檔案治理、日常採買與交通出行四大核心需求。主要成果如下：

1. **整合性平台**：四子系統共享 Flutter／Supabase 技術棧與高齡友善設計規範，長輩得以單一入口獲得多項服務，降低數位學習成本。
2. **混合式 AI 落地**：藥袋辨識（ML Kit + Gemini）、檔案語意搜尋（COCO-SSD + Sentence-BERT）、代購 NLU（規則優先 + 可選 Edge）均採輕量化、可解釋架構，適合資源有限的社區部署。
3. **志工協作閉環**：慢箋代領、物資代購、共乘媒合均整合志工／司機角色，符合偏鄉「一人多工、邻里互助」之實務。
4. **離線與即時並重**：物資子系統 Offline-First 設計與 Supabase Realtime 協作，在偏鄉不穩定網路下仍可維持可用體驗。

## 4.2 限制與挑戰

**技術層面：**

- 慢箋辨識成功率（約 85%）仍低於一般處方（約 92%）。
- 藥袋照片原圖尚未導入自動個資遮罩。
- 物資訂單級與品項級履行狀態尚未完全統一為單一狀態機。
- 檔案管理 AI 標籤目前以圖片為主，PDF／Word 內容理解待開發。

**研究方法層面：**

- 受限專題時程，缺乏正式長輩用戶測試與 SUS 量化數據。
- 系統尚未於實際社區關懷據點試點部署。
- 藥典收錄約 150 種常見藥品，覆蓋率有限。

## 4.3 未來展望

**短期（3 個月內）：**

- 導入藥袋影像自動個資遮罩；優化 Gemini 提示工程，提升慢箋辨識率至 90% 以上。
- 擴充 Flutter Integration Test 覆蓋核心流程。
- 整合衛福部食藥署開放藥品資料庫，擴充藥典。

**中期（6–12 個月）：**

- 物資子系統統一品項履行狀態機、接入 iOS APNs 推播。
- 智慧用藥新增家屬通知；檔案管理擴充文件語意標籤。
- 於苗栗縣社區關懷據點進行試點部署與可用性評估。

**長期（1–2 年）：**

- 串接長照 2.0 社區照顧關懷據點網絡，推廣為偏鄉標準化數位照護平台。
- 導入聯邦學習機制，在保護個資前提下持續優化藥袋辨識模型。
- 客家文化模組擴充為「口述歷史」數位保存平台。

本專題期望傳達：科技創新的價值不僅在於技術先進性，更在於能否回應社會真實需求、觸及最需要的群體。偏鄉高齡者應被積極納入設計過程，而非僅作為被動的服務對象。「明德 e 達人」正是此理念的具體實踐。
